GitHub 커밋을 블로그 글로 자동 변환하는 파이프라인 만들기

Webhook → Claude API → 자동 발행까지, 삽질 포함 전체 과정

E
레이더
2026.03.07 · 12 min read

코드를 쓰고 나면 그걸 정리해서 글로 쓰는 게 가장 귀찮은 일이었다. 커밋은 매일 하는데, 블로그 글은 한 달에 한 편도 힘들다.

그래서 만들었다. GitHub에 push하면 커밋 로그를 분석해서 블로그 글을 자동으로 생성하는 파이프라인.

왜 이걸 만들었나

개발자 블로그의 가장 큰 적은 귀찮음이다. 코드는 매일 쓰지만, 그걸 정리해서 글로 만드는 건 완전히 다른 에너지가 필요하다.

커밋 메시지에 이미 "뭘 했는지"가 다 들어있다. diff를 보면 "어떻게 했는지"도 알 수 있다. 이 데이터를 AI에게 넘기면 블로그 글의 80%는 자동으로 만들 수 있지 않을까?

3일 삽질한 결과, 된다. 근데 생각보다 까다로운 부분이 있었다.

전체 아키텍처

구조는 단순하다. GitHub Webhook으로 push 이벤트를 받고, Octokit으로 커밋 데이터를 수집하고, Claude API로 글을 생성한 다음 DB에 저장한다.

typescript
app.post('/api/webhooks/github', async (req, res) => {
  const { commits, repository } = req.body;
  
  const enriched = await Promise.all(
    commits.map(c => octokit.repos.getCommit({
      owner: repository.owner.login,
      repo: repository.name,
      ref: c.id,
    }))
  );

  const draft = await generatePost(enriched, repository);
  await db.posts.create({ data: draft });
});

이렇게 보면 쉬워 보이는데, 실제로는 커밋 데이터를 "글감"으로 가공하는 과정"이 핵심이었다.

삽질: diff 데이터 전처리

처음에는 커밋 diff 전체를 Claude API에 던졌다. 파일 하나 수정한 커밋은 괜찮은데, 리팩토링 커밋은 diff가 수백 줄이다.

typescript
// Before — diff 전체를 넘김 (토큰 폭발)
const prompt = `diff:\n${commit.diff}`;
// → 한 번에 $0.12

// After — 파일별 요약만 추출
const summary = commit.files.map(f => ({
  filename: f.filename,
  status: f.status,
  patch: f.patch?.slice(0, 500),
}));
// → 한 번에 $0.008

$0.12 vs $0.008. 같은 결과. 15배 차이.

AI 프롬프트 설계

글이 AI스러워 보이면 안 된다. 기존에 만들어둔 SKILL.md를 시스템 프롬프트로 넣고, 문체 규칙을 강제했다.

"모든 커밋에 교수를 부르지 마라." — 사소한 수정은 짧게, 의미 있는 변경만 깊게 써야 읽을 만한 글이 된다.
